# 同步与验证码

## 同步的实际调用层次

前端发起同步后，链路大致是：

1. 页面或弹窗触发同步
2. `appStore` 选择身份级或账号级同步动作
3. 调用 `src/services/tauri.ts`
4. 进入 `src-tauri/src/commands/sync.rs`
5. 再进入 `src-tauri/src/sync/mod.rs` 的 `BillSyncService`
6. 通过 `shmtu-cas` 完成登录、取账单、解析、入库

## 为什么会有这么多同步入口

当前前端区分了：

- 身份增量同步
- 身份全量同步
- 单账号增量同步
- 单账号全量同步

这不是重复设计，而是把用户场景拆开了。

## 同步服务职责

`BillSyncService` 负责：

- 找到可同步账号
- 根据配置决定验证码模式
- 处理手动验证码续传
- 执行多账号同步
- 管理同步锁
- 汇总进度与结果

## 验证码模式

当前配置层支持：

- `manual`
- `remote_ocr`
- `remote_ocr_http`
- `local_onnx`

程序不是把 OCR 写死在同步逻辑里，而是根据配置动态选路径。

## 为什么同步会停在“需要验证码”

在手动模式下，这就是设计目标，不是异常：

- 后端拿到 challenge
- 进度状态变成 `captcha_required`
- 前端显示验证码输入
- 用户继续输入后再续传

## 进度事件

同步进度通过事件推送回前端，命令层会把运行时进度转换成前端更容易用的结构。

这样前端能展示：

- 当前账号
- 当前页数
- 累计新增条数
- 当前阶段文字

## 本地 OCR

本地 OCR 的额外复杂度在于：

- 模型下载
- 模型完整性检查
- 模型加载与卸载
- 下载取消

因此它不仅是一个“识别模式”，还带了一套模型生命周期管理。
